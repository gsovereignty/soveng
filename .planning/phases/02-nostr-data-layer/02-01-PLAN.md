<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Public Nostr relays → data layer | Any author can publish arbitrary kind:30023 events; relay can send malformed/oversized events, unexpected tags, or never EOSE. All relay-supplied content is untrusted input (ASVS V5). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Denial of Service | useArticleFetch onevent flood | mitigate | Hard-freeze at 21 distinct coordinates + early `sub.close('freeze-at-21')` (D-02/D-05); frozen-flag guard ignores all further onevent calls — unbounded streamed events cannot grow state past 21. |
| T-02-02 | Denial of Service | Hanging/non-EOSE relay blocks resolution | mitigate | `maxWait: 8000` sets per-relay eoseTimeout; 9000ms backstop `setTimeout` force-closes; status resolves regardless (D-04). Rendering never depends on the timer (D-01). |
| T-02-03 | Tampering | Malformed kind:30023 event (missing/empty `d`, missing optional tags) | mitigate | parseArticle uses optional-chaining + `|| undefined` fallbacks (D-10), publishedAt falls back to created_at; onevent skips empty-`d` events to avoid degenerate coordinate collisions. No throw on any tag combination — unit-tested. |
| T-02-04 | Spoofing | Relay injecting wrong-kind or unsigned events | accept | SimplePool verifies every event signature before onevent fires and matchFilters drops non-matching kinds (nostr-tools built-in; RESEARCH Don't Hand-Roll). No custom verify needed. |
| T-02-05 | Tampering | Malicious `d`-tag content (injection/path-traversal chars) | accept | `d` is used only as a Map key + coordinate string; no filesystem/SQL/DOM sink in Phase 2. Low risk; Phase 3 must not interpolate it into HTML unsanitized (flagged downstream). |
| T-02-SC | Tampering | npm install of nostr-tools | accept | RESEARCH Package Legitimacy Audit: nostr-tools APPROVED (5-yr history, maintainer fiatjaf, official repo, no postinstall script). No [ASSUMED]/[SUS]/[SLOP] packages — no blocking checkpoint required. vitest is a mainstream dev dep. |

## Downstream (Phase 3/4) assumptions flagged
- `Article.content` is stored as a RAW Markdown string and is NEVER rendered in Phase 2. Phase 4 MUST sanitize (rehype-sanitize) before rendering — XSS surface is entirely a render-time concern.
- `Profile.picture` / `Article.image` URLs are author-controlled and may be `http:`, `javascript:`, or `data:` URIs. Phase 3 `<img>` rendering must guard (onError fallback; consider https upgrade). Phase 2 stores them verbatim and assumes nothing about safety.
</threat_model>
