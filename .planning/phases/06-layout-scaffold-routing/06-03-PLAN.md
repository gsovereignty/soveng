<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| URL hash → selectedNaddr → articleNaddr match | An attacker can craft any `#…` fragment; it flows into selectedNaddr and is compared (string equality) against articleNaddr() outputs |
| Relay article/profile data → ReadingPaneStub header | Untrusted title/pubkey/picture rendered in the header stub (text + Avatar) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-04 | Denial of Service | hash decode / selectedArticle match | mitigate | selectedNaddr is treated as an opaque string and only compared by equality to `articleNaddr(a)` outputs — a malformed/attacker hash simply matches nothing and falls through to the `[404]` state; no decode is invoked on the raw hash here (the `#` is stripped with `.slice(1)`), so no throw path exists |
| T-06-05 | Spoofing/Injection | ReadingPaneStub header | mitigate | The stub renders title/name/timestamp as React text children (auto-escaped) and the avatar via shadcn `AvatarImage` with an `AvatarFallback`; NO Markdown body and NO raw HTML are rendered this phase (D-06) — the XSS-relevant ArticleBody path is deferred to Phase 7 |
| T-06-06 | Tampering | npubEncode on untrusted pubkey | mitigate | npubEncode is wrapped in try/catch (03-01 discipline) so a malformed pubkey falls back to a hex slice instead of throwing |

Client-only static frontend. The only new untrusted-input surface is the URL hash, handled as an opaque equality-matched string (no eval, no decode-throw, no HTML). No backend, no secrets.
</threat_model>
