<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner to GitHub Pages | The CI job holds a GITHUB_TOKEN and publishes artifacts; over-broad permissions here are the main risk |
| public internet to GitHub Pages site | Anyone can fetch the static site; it is read-only with no user input this phase |
| npm registry to CI runner | npm ci pulls dependencies during the build (supply-chain surface) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Elevation of privilege | GitHub Actions workflow GITHUB_TOKEN scope | mitigate | Declare a least-privilege top-level permissions block: contents read, pages write, id-token write only. No write-all. Acceptance criteria assert these exact scopes. |
| T-02-02 | Denial of service | Concurrent Pages deploys | mitigate | concurrency group "pages" with cancel-in-progress true prevents overlapping/stuck deploys. |
| T-02-SC | Tampering | npm ci dependency install on CI runner | mitigate | npm ci installs strictly from the committed package-lock.json (reproducible, auditable). Deps are mainstream and pinned in STACK.md — no [ASSUMED]/[SUS]/[SLOP] packages to gate. |
| T-02-03 | Spoofing / Tampering | Third-party Actions (checkout, setup-node, upload-pages-artifact, deploy-pages) | accept | All actions are official GitHub-published actions pinned to a major version (v4/v3). Low risk; documented for awareness. Pinning to commit SHA is a possible future hardening, deferred. |
| T-02-04 | Information disclosure | Deployed static site content | accept | Site is intentionally public, read-only, no secrets in the bundle (zero-backend, no env secrets per project constraints). No PII. |
</threat_model>
