# Milestone v1.1: Local ML Content Filtering

**Status:** ✅ SHIPPED 2026-06-08
**Phases:** 5
**Total Plans:** 6

## Overview

A single phase that added in-browser ML content filtering to the v1.0 reader without
breaking its zero-backend, works-for-every-visitor property. Spam (transformers.js ONNX
classifier), non-English (franc-min language gate), and sub-500-word articles are hidden
from the list — all classification running off the main thread in a Web Worker, fail-open
throughout. Ships with full user-facing controls: an on-by-default toggle, a model
download progress indicator, a hidden-article count, and a spam-confidence slider that
re-thresholds stored scores instantly. Validated against real, deployed conditions on the
live `/soveng/` URL with a GO verdict pinning `SPAM_THRESHOLD = 0.90`.

Delivered: spam + language + length filtering for the Nostr long-form reader, entirely
in the visitor's browser, with transparent controls and no backend.

**Details:**
A synchronous foundation (Plan 01) established the `ClassificationLabel` 6-member union
with an allowlist-of-hides `isHidden` helper (unknown labels show by default), the
franc-min language gate (MIN_DETECT_CHARS=200, treats 'und' as English, strips code/Markdown
first, 20000-char ReDoS cap), and the 500-word `countWords` length gate. The ML
infrastructure (Plan 02) put the transformers.js ONNX spam classifier in a Web Worker as a
module-level singleton (mirroring the pool.ts pattern), with `numThreads=1` and a
`wasmPaths` CDN URL pinned to the exact onnxruntime-web transitive version
(`1.26.0-dev.20260416-b7804b056c`) verified by a `check:ort-version` script. The
`useClassification` hook (Plan 03) orchestrated cheap-gates-first, cached raw ONNX scores
per event id (enabling instant label recompute on threshold change without re-inference),
and failed open on every error path. The controls UI (Plan 04) used only shadcn primitives —
toggle, 0.50–0.99 slider, download progress, hidden-count badge, model-failure notice.
Integration (Plan 05) inserted a single `visibleArticles` memo between `sortedArticles` and
the three downstream memos, with a default-on localStorage-persisted toggle. Validation
(Plan 06) added DEV-only spam-score logging (tree-shaken from production), deployed to the
live URL, and confirmed legitimate Bitcoin/Lightning/Nostr articles were not over-filtered
at 0.90 — recording a GO verdict.

## Milestone Summary

**Scope Changes:**

- Former **Phase 6 (Filter Controls)** and **Phase 7 (False-Positive Recovery)** were folded
  into Phase 5 on 2026-06-08 during phase discussion (commit 700a576). v1.1 shipped as a
  single phase delivering the full filtering feature plus its controls.

**Key Decisions:**

- `SPAM_THRESHOLD = 0.90` (not the SMS-model default 0.50) — domain shift from SMS training
  data to crypto/Nostr long-form; pinned as a named constant with a validation-date comment
  after the live GO verdict (D-01).
- Worker singleton at module level (mirrors `pool.ts`), never inside `useEffect` — StrictMode-safe.
- `?worker` / module-singleton ONNX pipeline with `numThreads=1` + version-pinned `wasmPaths`
  CDN URL — required for the GitHub Pages `/soveng/` base path (no SharedArrayBuffer, no .wasm 404s).
- Cheap-gates-first pipeline (franc language gate → 500-word length gate → ONNX worker) —
  avoids spinning up inference for articles cut by a cheap synchronous check.
- Raw scores cached per event id separately from labels, so the slider re-thresholds instantly
  without re-running inference (D-02).
- `visibleArticles` as the single ML-filter insertion point — facets/dynamicCounts/filteredArticles
  all derive from the post-ML view; integration was surgical, one memo.
- `isHidden` is an allowlist-of-hides (spam, non-english, short only) — fail-open by construction.
- CTRL-06 reinterpreted: no separate "show hidden" control — the spam-confidence slider at its
  maximum effectively disables the spam filter, surfacing any false positives.

**Issues Resolved:**

- All 5 critical ML-on-GitHub-Pages pitfalls (wasm 404s, SharedArrayBuffer warning, ORT version
  drift, over-filtering at default threshold, main-thread jank) resolved and verified on the
  live deployed URL — not just localhost.
- tsc type errors in useClassification tests surfaced post-merge (3980105) — resolved.
- Cross-platform native bindings in package-lock.json restored (3c5b62d).

**Issues Deferred:**

- **SPAM-05**: Replace the SMS-trained spam model with a Nostr/long-form-trained classifier if
  false-positive rates stay high after launch feedback.
- **SPAM-06**: Per-article "why filtered" disclosure (spam vs language + confidence).
- **MUTE-01**: Pubkey denylist / mute list.
- Carried from v1.0: **DATA-02** requirement/constraint text ("21 most recent, newest first")
  still diverges from shipped behavior (uncapped, reply-count sort via 260607-vqt) — reconcile
  in a future milestone.

**Technical Debt Incurred:**

- Spam model is a general SMS-trained classifier, not domain-tuned for crypto/Nostr long-form;
  the 0.90 threshold is the mitigation. Domain-tuned model deferred (SPAM-05).
- The 500-word `MIN_WORDS` boundary is an editorial guess; may need tuning against reader feedback.
